# PicaFlux — 产品方案

> **PicaFlux**: 面向 AI 时代的开源创意素材中枢
>
> "pica" (像素/图像) + "flux" (流动/变换) — 让素材在创意工作流中自由流转

---

## 一、产品定位与愿景

### 1.1 一句话定义

PicaFlux 是一个**开源的、AI 原生的创意素材管理与处理平台**，统一管理图片、视频、3D 模型、音频等多模态素材，并通过 MCP/ACP 协议连接 AI 工具链，让创意工作者从"素材混乱"中解放出来。

### 1.2 核心愿景

```
传统 DAM:  素材仓库 → 存储 → 检索 → 下载
PicaFlux:  素材中枢 → 管理 → 处理 → AI增强 → 工作流串联 → 协作分发
```

我们不只是做一个"更好的 Eagle"，而是构建 **AI 时代的创意素材操作系统**：

- **素材管理** — 不只是存储，而是理解（AI 自动标签、语义搜索）
- **素材处理** — 不只是预览，而是原地处理（裁剪、格式转换、批处理）
- **AI 工作流** — 不只是生成，而是管理（Prompt→生成物→迭代的完整链路）
- **工具连接** — 不只是孤岛，而是中枢（MCP/ACP 连接 Photoshop、Blender、ComfyUI 等）

### 1.3 目标用户

| 用户群体              | 核心痛点                                     | PicaFlux 价值                     |
| --------------------- | -------------------------------------------- | --------------------------------- |
| **AI 创作者**         | AIGC 产出量大，prompt 与生成物脱节，无法回溯 | Prompt-Asset 关联管理，工作流追溯 |
| **设计师/插画师**     | 素材分散在多个工具，格式转换繁琐             | 统一管理 + 批量处理 + 工具桥接    |
| **3D 艺术家**         | 模型/贴图/材质管理混乱，预览困难             | 3D 原生支持，实时预览，资产依赖图 |
| **游戏开发者**        | 素材管线复杂，版本管理困难                   | 版本控制 + Pipeline 集成          |
| **独立开发者/小团队** | 买不起或不需要企业级 DAM                     | 开源免费，本地优先                |

### 2.3 PicaFlux 六大差异化优势

#### 差异 1: Prompt-Asset 关联管理（市面空白）

现有工具要么管素材（Eagle），要么管 Prompt（Montage/Versuno），**没有人把两者关联起来**。

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Prompt v1  │────→│  Image A    │     │  Image B    │
│  "a cat..." │     │  seed:1234  │     │  seed:5678  │
└──────┬──────┘     └─────────────┘     └─────────────┘
       │
       ▼
┌─────────────┐     ┌─────────────┐
│  Prompt v2  │────→│  Image C    │
│  "a cute.." │     │  upscaled   │
└─────────────┘     └──────┬──────┘
                           │ derived from
                           ▼
                    ┌─────────────┐
                    │  Video D    │
                    │  img2video  │
                    └─────────────┘
```

PicaFlux 记录完整的 **创意谱系（Creative Lineage）**：

- Prompt 版本历史
- 每个 Prompt 对应的生成参数（模型、seed、步数等）
- 生成物之间的派生关系（原图→放大→视频→3D）
- 支持从任意节点"重新生成"或"分支迭代"

#### 差异 2: MCP/ACP 原生协议桥接（首创）

不是简单的"导出到 Photoshop"，而是通过标准协议实现**双向实时桥接**：

```
PicaFlux ←─ MCP ──→ Photoshop (通过 MCP Server)
PicaFlux ←─ MCP ──→ Blender
PicaFlux ←─ MCP ──→ ComfyUI / Stable Diffusion
PicaFlux ←─ MCP ──→ Figma
PicaFlux ←─ ACP ──→ AI Agent（自动处理素材的智能体）
```

用户场景示例：

- 在 PicaFlux 中选中一张图，右键"在 ComfyUI 中放大"，结果自动回流
- AI Agent 通过 ACP 自动执行"将所有 PNG 转为 WebP 并压缩到 500KB 以下"
- Blender 中渲染完成的模型自动归档到 PicaFlux 对应项目

#### 差异 3: 多模态素材统一处理引擎

不只是"能打开各种格式"，而是**内建处理能力**：

| 素材类型 | 管理 | 预览              | 处理能力                                |
| -------- | ---- | ----------------- | --------------------------------------- |
| 图片     | ✅   | ✅ 含 PSD/AI 图层 | 裁剪、缩放、格式转换、滤镜、去背景      |
| 视频     | ✅   | ✅ 时间线预览     | 截帧、裁剪、转码、抽取音频、GIF生成     |
| 3D 模型  | ✅   | ✅ WebGL 实时渲染 | 格式转换、LOD生成、材质预览、缩略图渲染 |
| 音频     | ✅   | ✅ 波形图         | 裁剪、格式转换、降噪                    |
| 字体     | ✅   | ✅ 字符预览       | 子集化、格式转换                        |
| Prompt   | ✅   | ✅ 模板预览       | 版本管理、变量替换、批量生成            |

#### 差异 4: 非侵入式 + Git-like 版本管理

```
Eagle:     复制原文件 → 占用双倍空间 → 迁移困难
Billfish:  索引模式 → 不复制 → 但无版本管理
PicaFlux:  索引模式 + Content-Addressable Storage + 版本快照
```

- **默认索引模式**：不移动/复制原文件，零额外空间
- **可选托管模式**：Content-Addressable Storage (CAS)，相同内容自动去重
- **版本快照**：类 Git 的素材版本管理，支持 diff 预览
- **导出自由**：随时导出为纯文件夹结构，无厂商锁定

#### 差异 5: 可编排的批处理管线（Pipeline）

```yaml
# pipeline.yaml - PicaFlux 批处理管线示例
name: 'AI 生成图后处理'
trigger:
  folder_watch: './ai-output'

steps:
  - action: auto_tag
    model: 'clip-vit-large'

  - action: upscale
    model: 'real-esrgan-x4'
    condition: 'width < 2048'

  - action: remove_background
    condition: "tags contains 'character'"

  - action: convert
    format: webp
    quality: 90

  - action: organize
    rule: '/{year}/{month}/{tags[0]}/'
```

#### 差异 6: 插件优先的架构（VS Code 式扩展模型）

借鉴 VS Code 的插件体系，插件用 TypeScript 编写，开发者零学习成本：

```

│              PicaFlux Core (Main Process)        │
```

## 三、核心功能模块

### 3.1 素材库（Library）

- **智能收藏夹**：基于规则自动分类（类 macOS 智能文件夹）
- **多维标签系统**：层级标签 + 颜色标签 + AI 自动标签
- **语义搜索**：基于 CLIP/多模态模型，用自然语言搜索素材
- **相似素材发现**：基于视觉嵌入的相似图搜索
- **素材关系图谱**：可视化素材间的派生、引用、组合关系
- **收藏与评分**：快速标记和筛选优质素材

### 3.2 处理引擎（Processor）

- **非破坏性编辑**：所有操作可撤销、可回溯
- **批量处理**：选中多个素材一键应用相同操作
- **管线编排**：可视化拖拽编排处理步骤（类 Node 编辑器）
- **格式工厂**：支持 100+ 格式互转
- **AI 处理器**：
  - 智能去背景（SAM / RMBG）
  - 超分辨率放大（Real-ESRGAN）
  - 风格迁移
  - 图片修复（Inpainting）
  - 视频插帧

### 3.3 AI 工作台（AI Workbench）

- **Prompt 编辑器**：
  - 富文本 Prompt 编辑，支持变量和模板
  - 版本对比（diff view）
  - Prompt 库：个人收藏 + 社区分享
- **生成管理**：
  - 一键发送到 ComfyUI / SD WebUI / Midjourney 等
  - 自动捕获生成参数（metadata 解析）
  - 批量生成 + 参数网格（Grid）对比
- **迭代追踪**：
  - 从生成物一键查看使用的 Prompt 和参数
  - 创意谱系树（Lineage Tree）可视化
  - 支持"从这里分支"继续迭代

### 3.4 连接器（Connectors）

- **MCP Server**：PicaFlux 作为 MCP Server，允许 AI Agent 操作素材库
- **MCP Client**：PicaFlux 连接外部 MCP Server（Photoshop、Blender 等）
- **ACP Bridge**：Agent 间协作，如"自动化素材处理智能体"
- **浏览器扩展**：一键收藏网页素材（类 Eagle）
- **文件系统监控**：监控指定文件夹，自动导入新素材
- **云存储同步**：可选对接 S3/OSS/本地 NAS

### 3.5 工作区（Workspace）

- **项目隔离**：每个项目独立的素材库和配置
- **看板视图**：类 Trello 的素材状态管理（待处理→处理中→已完成）
- **协作支持**：（远期）多人协作编辑和素材共享

### 4.2 技术选型

| 层级     | 选型                              | 理由                                                                                                  |
| -------- | --------------------------------- | ----------------------------------------------------------------------------------------------------- |
| 桌面框架 | **Electron**                      | 生态成熟，VS Code/Figma 验证可行，开发效率高                                                          |
| 构建工具 | **Vite + vite-plugin-electron**   | 快速 HMR，主进程/预加载/渲染统一构建（与 npm 包 `electron-vite` CLI 不同属，同属 Electron+Vite 生态） |
| 前端     | **React + TypeScript**            | 生态最大，社区贡献者友好                                                                              |
| UI 组件  | **TailwindCSS + Radix UI**        | 美观、可访问、无样式约束                                                                              |
| 后端语言 | **Node.js (TypeScript)**          | 与前端同语言，开发效率高，npm 生态丰富                                                                |
| 数据库   | **better-sqlite3 + FTS5**         | 本地优先，同步 API 性能好，全文搜索内建                                                               |
| 向量搜索 | **sqlite-vec / usearch**          | 语义搜索向量索引，嵌入 SQLite 零部署                                                                  |
| 图片处理 | **sharp**                         | libvips 驱动，性能极好，API 成熟                                                                      |
| 视频处理 | **fluent-ffmpeg + ffmpeg 二进制** | 业界标准，格式支持最全                                                                                |
| AI 推理  | **onnxruntime-node**              | 本地推理 CLIP/SAM 等模型，支持 CPU/GPU                                                                |
| 3D 渲染  | **Three.js**                      | JS 生态 3D 标准，GLB/GLTF/OBJ 支持好                                                                  |
| MCP 协议 | **@modelcontextprotocol/sdk**     | 官方 TypeScript SDK，文档最全，维护最活跃                                                             |
| 文件监控 | **chokidar**                      | 跨平台文件监控，成熟稳定                                                                              |
| 插件系统 | **Node.js 模块 + iframe 沙箱**    | 开发者零学习成本，TypeScript 直接写插件                                                               |
| 性能优化 | **napi-rs（渐进式）**             | 后期将热路径用 Rust 重写为 native addon                                                               |

### 4.3 渐进式性能优化策略

采用"先跑通，再优化"的务实路线：

```
Phase 0-2:  100% TypeScript/Node.js
            ↓ 够用，快速迭代
Phase 3:    识别性能瓶颈（profiling）
            ↓ 数据驱动决策
Phase 3-4:  用 napi-rs 将热路径重写为 Rust native addon
            - 批量文件扫描与元数据提取
            - 内容哈希计算 (SHA-256)
            - 图片缩略图批量生成
            - 向量相似度计算
```

这个策略的好处：

- **前期**：全 TypeScript 栈，开发快，贡献者友好
- **后期**：只在真正有瓶颈的地方引入 Rust，投入产出比最高
- **对外透明**：napi-rs 编译为 .node 文件，使用者无需安装 Rust
